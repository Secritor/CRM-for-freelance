# Тестирование (unit)

В проекте используются **Vitest** и **React Testing Library** для юнит-тестов компонентов и утилит.

## Запуск

- `npm run test` — watch-режим (перезапуск при изменении файлов)
- `npm run test:run` — один прогон (удобно для CI)

## На что обращать внимание при написании тестов

### 1. Утилиты и чистые функции (`src/lib/*.ts`)

- Тестировать в первую очередь: предсказуемый ввод/вывод, граничные случаи.
- Для дат использовать `vi.useFakeTimers()` / `vi.setSystemTime()`, чтобы не зависеть от «сейчас».
- Проверять невалидный ввод (например, `formatDisplayDate('invalid')` → `'—'`).

### 2. Презентационные компоненты (Title, EmptyState, CardList)

- Рендер: наличие заголовков, текста, кнопок в документе.
- Поведение: клик по кнопке вызывает переданный callback (`onButtonClick`).
- Опциональные пропсы: кнопка не рендерится без `buttonText`; `className` применяется к корневому элементу.

### 3. Компоненты с Redux

- Использовать `render` из `@/test/test-utils` (уже обёрнут в `<Provider store={store}>`).
- При необходимости подготовить стейт: `store.dispatch(...)` перед `render`, либо тестировать с дефолтным стейтом.
- Проверять отображение данных из стейта и реакции на действия (диспатч после клика, обновление UI).

### 4. Компоненты с роутером (React Router)

- Обернуть в `<MemoryRouter>` или добавить в `test-utils` опциональный wrapper с `MemoryRouter` и при необходимости с `Routes/Route`.
- Проверять переходы по ссылкам и отображение контента в зависимости от пути.

### 5. Модалки и формы (UniversalModal, LoginPage)

- Рендер по `open={true}`; проверка наличия полей и кнопок.
- Сабмит формы: вызвать submit, проверить вызов `onSubmit` с ожидаемыми данными.
- Валидация: неверный ввод → сообщение об ошибке или блокировка submit.

### 6. Общие принципы

- Искать элементы по **ролям и тексту** (`getByRole`, `getByLabelText`, `getByText`), а не по классам или тест-ид, если возможно.
- `userEvent` для имитации кликов и ввода (предпочтительнее `fireEvent`).
- Один тест — один сценарий; несколько проверок в одном тесте допустимы, если они относятся к одному сценарию.
- Не тестировать детали реализации (внутренний стейт, классы), тестировать наблюдаемое поведение и контент.

## Структура

- `src/test/setup.ts` — подключение `@testing-library/jest-dom` (матчеры вроде `toBeInTheDocument()`).
- `src/test/test-utils.tsx` — кастомный `render` с Redux Provider; импорт `screen`, `userEvent` и т.д. из `@/test/test-utils`.
- Тесты рядом с кодом: не используются. Все тесты лежат в **src/test/unit/** (структура повторяет src: unit/lib/, unit/app/components/ и т.д.).

## Примеры

- `src/lib/dateUtils.test.ts` — тесты утилит дат.
- `src/app/components/Title.test.tsx` — рендер и клик по кнопке.
- `src/app/components/EmptyState.test.tsx` — рендер контента и опциональный `className`.
